# 面试回答稿 · 三大终极反问

> **基础**：来自 2026-06-18 外部 Opus 级评审的三大反问（详见 `docs/external-review-2026-06-18.md` §5）
> **使用方式**：每题先背一句话核心回答，再按"展开 + 钩子"自然延伸
> **核心策略**：不辩护、不逃避、用诚实换信任，把每个短板都讲成「我知道，且我已经在解决」

## Q1 · 关于 BM25 是否够用

> **面试官提问**：「你的产品宪法说证据优于推测，但你对'BM25 已够'这个核心架构假设零实测数据。你凭什么相信 BM25 够用？」

### 一句话核心回答

> 「**坦白说我不知道 BM25 是不是真够用——但我知道这件事我不能假装知道，所以我已经设计好了证伪标准和实验，下一步就是跑数据。**」

### 展开

「这个问题我得先认错，再讲设计。

**认错**：我之前在 README 和文档里写过 'FTS5+BM25 已够用'。这句话严格说是推测，违反了我自己写在 .clinerules 里的第一条宪法——证据优于推测。如果一个面试官追问到这里，我没有数据反驳，我也不应该反驳。

**设计**：但我在做这个项目的过程里给自己定了一条规矩——**任何技术选型都必须有可证伪的判定边界**。所以在 dogfooding sprint 里，我已经规划了一个量化实验：

- **corpus**：把我自己 14 天 sprint 产生的真实 memory 当语料，合成扩展到 10 万条
- **标注**：50-100 个 (query, 期望命中 doc_id) 对，自己人工标
- **对照组**：纯 BM25 / +Node 内置 Intl.Segmenter 中文分词 / +停用词过滤
- **指标**：recall@5、recall@10、p50/p99 延迟
- **判定线**：如果 recall@10 < 70% 或 p99 > 100ms，触发重评估——比如加入 SQLite FTS5 的 trigram tokenizer 或重新讨论是否破例引入向量。

所以我对 BM25 的态度不是'我相信它够'，而是 '**我知道它在哪个边界以内够，越过边界我会重评估**'。这个设计本身比答案重要。」

### 钩子（引导面试官进下一个话题）

> 「这其实暴露了我项目里的一个普遍模式：**我做的每个'不做'决策，都应该附带一个可证伪的边界条件**——但我目前只有一两个做到了这点，剩下的还没补全。这是我接下来要做的工作。」

→ 自然引到 Q3（克制叙事）

## Q2 · 关于 handoff 进 git 但 memory 全本地

> **面试官提问**：「你判定项目状态该和代码同生命周期、可 review、可恢复，所以 handoff 进 git。但你把 memory 全留在本地 ~/.cline-skills/，且 project_hash 绑死绝对路径。这两个决策的哲学是矛盾的——怎么自圆其说？」

### 一句话核心回答

> 「**这不是自圆其说，这是真矛盾，是我目前的设计缺陷。修复路径已经明确：semantic 和 procedural 这两类记忆应该跟 git 走，episodic 留本地，绑定键从绝对路径换成 project_id。**」

### 展开

「这个问题问得很狠，我承认它是我项目里最大的哲学不一致。

**先说清三种 memory**：
- `episodic`（具体事件，比如'第 N 次会话试了 Y 失败了'）：噪音大、量大、个人化——**留本地是对的**
- `semantic`（项目事实，比如'本仓库用 node:sqlite 不用 pgvector'）：项目级共享知识——**应该进 git**
- `procedural`（流程/规范，比如'部署前必须跑 validate-skills.js'）：团队/未来的自己都需要——**应该进 git**

我现在所有 kind 都进本地 SQLite，**这违反了我自己'handoff 进 git'的逻辑**。一个一致的设计应该是：handoff 进 git，semantic/procedural 也进 git。

**修复路径**：
1. 加 `memory_export` 把 semantic/procedural 导出成 JSONL 进 git，可 diff 可 review
2. 加 `memory_import` 在新机器 `git clone` 后重建本地 SQLite
3. project_hash 改成按 `git config --get remote.origin.url` 优先，回退到 `.cline/project_id`（进 git 的 UUID），最后才回退绝对路径——彻底解决跨设备断档

**为什么我现在还没做？** 因为我刚刚发现这个不一致——是 6 月 18 号外部评审推过来的。我把它列进了 sprint 后的 P0 修复清单。**我把它当成现在要修的 bug，不是要辩护的设计**。」

### 钩子

> 「其实这件事让我意识到一件更深的事：**项目的'哲学一致性'比任何单点功能都重要**。如果 handoff 走 git 是对的，那它的逻辑就应该一以贯之，否则就是漏洞。我下一阶段最有信心的工作就是把这个漏洞补完——这是我'状态恢复'叙事真正闭环的关键。」

→ 把"我还没做的事"讲成"清晰的下一步"，反而显得有路线图

## Q3 · 玩具论

> **面试官提问**：「这看起来就是几个 Markdown 模板加一个 SQLite 包装，凭什么算 infra 能力？」

### 一句话核心回答

> 「**对，从功能维度看就是 Markdown + SQLite。但 infra 能力不在'我做了什么'，而在'我克制着没做什么、每个不做都能用一句话讲出可证伪的理由'。**」

### 展开

「这个问题我接受。论功能复杂度，我没有向量库、没有分布式、没有 LLM 调用——确实就是 Markdown 模板 + SQLite + MCP 协议封装。

**但请允许我反问一个 infra 问题**：一个项目最难的是什么？是写功能，还是**在合理的诱惑下不写功能**？

我觉得我在这个项目里**最值钱的能力**不是写了什么，而是面对每一个'要不要加 X'的诱惑时，能不能给出**可证伪的拒绝理由**。让我具体讲三个：

**例子 1：为什么不做向量库？**
- 不是因为我不会做，而是：FTS5 + BM25 在 10 万条以内的项目级私有 memory 上，**召回率衰减点比向量库的引入成本来得晚**
- 引入向量库的代价：额外原生编译依赖（node-gyp 链）+ embedding API 调用费用 + 模型版本一致性问题——违反我们「零额外原生编译依赖」的产品宪法
- **触发线**：recall@10 < 70% 或用户反馈跨语言/同义词漏召率高，再做评估
- 这是一个**有边界的、可被证伪的**'不做'决策

**例子 2：为什么不在 server 内调 LLM 做摘要？**
- 因为 server 必须保持纯逻辑、可单元测试、可离线、可重放
- LLM 调用应该在 client（Cline）侧由 SKILL.md 引导
- **架构边界**：所有不确定性挪到 client，server 永远确定
- 这是**软件工程基本功**，不是新概念

**例子 3：为什么 single-active-handoff per branch？**
- 因为 git 的并行任务**就该**用不同 branch
- 如果用户在同 branch 上跑多个独立任务，那是反 git 工作流，我不该为反模式付出复杂度成本
- 但我承认有边角场景（trunk-based 单 main）会撞——**已列入'已知约束'，等真撞上再说**
- 这符合我宪法第三条：**复杂度必须被证明，不要为假想需求建抽象**

**这三个'不做'**——能不能讲出可证伪的理由，比能不能再加三个功能更能体现我懂不懂 infra。一个真正的 infra 工程师做的最难的事，不是把系统做大，而是**让系统在边界以内做正确的事，并且承认边界存在**。」

### 钩子（杀招）

> 「如果您觉得这套'克制叙事'听起来像是给单薄项目的辩护，那我接受这个判断。但我希望你看一件事——我的 .clinerules 里有三条核心宪法：证据优于推测、问题定义优于方案设计、复杂度必须被证明。**这三条不是为这个项目写的，是我的工程价值观——这个项目只是它们的第一次落地实践。** 我希望面试聊的不只是这个项目，而是这种思考方式可以怎么用在你们的真实场景里。」

→ 把面试从「拷问项目」拉到「价值观对齐」，这是应届生最该用的高阶话术

## 通用面试技巧（用上面三题练出来的模式）

| 模式 | 适用场景 |
|---|---|
| **先认错再设计** | 任何"你为什么不做 X"或"你怎么知道 Y" |
| **承认 + 路线图** | 任何指出真缺陷的问题（不要硬辩） |
| **从功能维度跳到决策维度** | 当面试官说"这就是个 X"时（Q3 模式） |
| **每个'不做'附带可证伪边界** | 整个项目的核心叙事，重复使用 |
| **把价值观当锚** | 收尾时把项目讲成价值观的实践，而非孤立成果 |
