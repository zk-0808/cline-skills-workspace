## 二、使用过程中的常见问题

### FAQ: 调查工具本身可能是卡死嫌疑人（自我指涉陷阱）

**现象**：你在调查一个"命令为什么卡死"的问题，于是用同样的工具执行诊断命令——结果诊断命令也卡死了。你不知道是"诊断命令有问题"还是"原问题复现了"。

**根因**：工具链（Shell / 管道 / 编码）的某些组合本身就是卡死触发条件。当调查工具与被调查对象共享同一运行时，诊断行为可能触发同一类故障。

**解法**：
1. 第一时间换一个**完全不同的执行路径**——如换 Shell（`cmd /c`）、换运行时（`node -e`）、或直接用文件工具（`read_file` / `list_files`）
2. 不要在"卡死嫌疑犯"上跑同样的管道/编码模式来诊断它
3. 调查卡死问题本身就需要**卡死安全的诊断工具链**

**案例**：详见 `DEV_NOTES.md`「2026-06-20 凌晨: PowerShell 7 管道传递 emoji 数据导致终端挂起」——本机 GBK 936 + PS 管道 + emoji 输出导致挂起的具体复现与修复。

### 时间锚混淆：用"当前快照"否定"历史推断"

**现象**：调查过程中环境被修改（用户改了文件、并发进程动了状态），但你把"t1 时刻读到的状态"等同于"用户提问时 t0 的状态"，导致原本正确的归因被推翻。

**根因**：证据本身正确，但时间锚错了。文件系统不暴露"上次修改原因"，`disabled: true` 这类状态没有时间信息。

**解法**：
1. 看到反证立刻翻转结论前，先问"原结论是基于什么时刻的证据？现在的证据是什么时刻的？"
2. 用户在对话中说"我刚改了 X" / "我刚关了 Y" 是高价值信号，应主动询问而非默认环境静止
3. 调查类任务可在 thinking 中维护一个"证据时间线"，而非只记"证据是什么"

**案例**：详见 `DEV_NOTES.md`「2026-06-18: 证据没有时间戳——用"当前快照"否定了"历史推断"」

### 用户质疑是发现盲点的最好方式

**现象**：Agent 容易太快相信第一个看起来合理的解释，用户/评审者的质疑往往能指向被忽略的维度。

**解法**：
1. 不确定时优先搜索社区方案，为问题解决拓宽思路，不要闭门造车
2. 推测必须明确标注 `[未验证]`，不得编造（已在宪法一落地）
3. 架构变更前先做最小验证实验（已在宪法三落地）
4. "分析"关键字不应自动归于 research，先看目标语义（已在规范 5 落地）

**案例**：详见 `DEV_NOTES.md`「2024-06-17: 用户建议的价值」「2024-06-17: 不确定时搜索社区方案」

### 调查假设池必须包含"非业务代码路径"

**现象**：调查"文件被删/状态被改"类问题时，只盯着业务代码（handler/库函数），忽略了测试套件、CI 脚本、git hook、IDE 自动保存等也是同一份代码的执行路径。

**解法**：
1. 根因调查的假设池必须包含"非业务代码路径"——测试、CI 脚本、git hook、IDE 自动保存等
2. 写测试时强制问"如果开发者跑这个测试 100 次，工作树会变什么样？"
3. 看到测试输出有"工作目录：<真实路径>"字样，应立即质疑测试隔离性

**案例**：详见 `DEV_NOTES.md`「2026-06-19 夜·续: handoff 测试套件污染真实工作树」

## 五、事故复盘记录

### 2026-06-20: 超长内联命令 + 终端虚响应 + 编辑器空白文件 = 三次写入失败

- **类型**：错误（系统性问题，已触发 .clinerules 规则补丁）
- **背景**：在 `chore/v1.0-freeze-p0` 分支上需要写入 `test-isolation.js` 文件来验证手写 memory/handoff 路径隔离
- **问题**：三次写入尝试均失败

**第一次尝试**：用 `node -e "..."` 内联命令（>2KB）通过 PowerShell 写入文件 → 终端卡死无输出，但返回假成功信号（The command's output could not be captured）。

**第二次尝试**：用 `node -e "console.log('probe')"` 诊断终端是否存活 → 同样卡死（自我指涉陷阱——诊断工具与被诊断对象共享同一运行时和管道编码路径）。

**第三次尝试**：绕过终端，用 `write_to_file` 新建文件 → VS Code 在编辑器面板打开空白文档，用户关闭文档时未保存，文件实际未写入磁盘。

- **Q0（系统性 vs 偶然）**：系统性问题。三个机制独立触发——PSReadLine 缓冲区溢出、终端虚响应假成功、write_to_file 新建文件依赖编辑器保存。任何项目在 Windows 中文系统 + PowerShell 7.6 + 大量 CJK 内容写入时都可能复现。

- **Q1（错在哪）**：三个层次：
  1. 使用 >2KB 的 `node -e` 内联命令写入 Markdown
  2. 终端假死后用 node -e 诊断（而非换 `cmd /c` 或文件工具）
  3. 写新文件时未预判编辑器打开+关闭行为

- **Q2（为什么错）**：
  1. 低估了 PowerShell + 超长内联命令 + CJK 字符组合的风险（PSReadLine 缓冲区溢出是已知 PS 7.6 问题）
  2. 思维盲点：诊断工具与被诊断对象共享同一运行时，未考虑"自我指涉"
  3. 对 write_to_file 的工具行为假设错误——认为"新建文件"是纯后端操作，实际 VS Code 集成会打开编辑器

- **Q3（现有规则为何没阻止）**：
  1. 规范 10 在事故发生前只有 5 条陷阱，缺「超长内联命令」条目
  2. 规范 1 有「环境变更前验证可观测性」，但未覆盖「终端虚响应后如何安全重探」
  3. 无规则约束 write_to_file 新建文件的行为

- **Q4（规则调整）**：
  1. 规范 10 新增第 6 条：超长内联命令陷阱（>2KB 的 `node -e` → 改写入独立 .mjs 文件）
  2. 规范 1 新增终端虚响应检测：短命令探针 + read_file 回退 + `cmd /c` 兜底
  3. 规范 10 末尾新增：链式命令拆分 + 终端假死后换 `cmd /c` 或文件工具
  4. 规范 11 新增：工作目录边界（Platform ≠ User Project）

- **Q5（未来如何更早发现）**：
  1. 看到 `The command's output could not be captured` 时立刻触发终端虚响应检测流程
  2. 任何 >200 字符的 `node -e` 都先问"能不能拆成独立 .mjs 文件"
  3. write_to_file 新建文件后，用 read_file 确认文件实际落盘内容

- **关联规则**：.clinerules 规范 1（终端虚响应检测）、规范 10（超长内联命令陷阱 #6）、规范 11（工作目录边界）
- **来源**：2026-06-20 chore/v1.0-freeze-p0 会话实测
